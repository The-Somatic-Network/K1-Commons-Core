# ADR-001: AI as an Accessibility Tool

**Date:** 2026-06-22
**Status:** proposed
**Context:**
The project aims to build an open-source cooperative freelance marketplace. As the project scales, there will be increasing pressure to use AI for code generation, auto-completion, and automated documentation. However, the project's core values prioritize human agency, worker sovereignty, and anti-exploitation. We need a clear decision on the boundaries of AI usage within this project.

**Decision:**
AI is permitted strictly as an accessibility and efficiency tool. Self-hosted and locally-run AI models are fully encouraged. The following rules apply:

1. **Allowed uses:** Summarizing technical brainstorming threads, formatting documentation, and general accessibility/efficiency tasks.
2. **Prohibited uses:** Automated code generation ("slop"), AI-driven code overrides, and any AI tooling that replaces human decision-making in the codebase.
3. **Deployment model:** AI must run self-hosted and locally (e.g., a friend's self-hosted Qwen model). This ensures full privacy, local control, and alignment with the project's anti-exploitation values. Cloud-hosted AI services (OpenAI, Anthropic API, etc.) are not permitted for code-related tasks.

**Consequences:**
- ✅ Preserves human ownership of the codebase and documentation.
- ✅ Self-hosted AI keeps data private and under community control.
- ✅ Aligns with cooperative values — no external AI companies profiting from our labor.
- ⚠️ May slow down rapid prototyping compared to cloud AI tools, but ensures ethical consistency.
